### Часть 2. Полное Техническое Задание «ПроМастер»

#### 2.1. Стек технологий (Производительность и Надежда на Highload)

Ты просил «самый крутой стек на Python/Django». Вот конфигурация, которая позволяет держать 10к RPS на среднем железе, если руки из плеч.

| Компонент | Технология | Обоснование выбора |
| :-- | :-- | :-- |
| **Бэкенд-фреймворк** | **Django 5.x** + **Django REST Framework (DRF)** | Быстрота разработки админки + мощный API для будущих мобильных приложений. |
| **Асинхронные задачи** | **Celery** + **Redis** (как брокер) | Отправка SMS/Email уведомлений, генерация отчетов, тяжелые выборки по каталогу СТО **не должны** вешать основной поток запроса. |
| **База данных** | **PostgreSQL 16** с расширением **PostGIS** | PostGIS нужен для геопоиска «Найти СТО рядом со мной». Без него твой поиск по координатам будет адски тормозить. |
| **Кэширование** | **Redis** (также как кэш-сервер) + **Django Cache Framework** | Кэширование страниц СТО и результатов поиска. Снизит нагрузку на базу в 100 раз. |
| **Веб-сервер** | **Nginx** (статика) + **Gunicorn/Uvicorn** | Uvicorn с workers для асинхронности (ASGI), так как чаты будут на WebSockets. |
| **Фронтенд** | **Vue.js 3 (Composition API)** + **Vite** + **Tailwind CSS** | Ты хочешь «чтобы не тормозило». SPA на Vue даст ощущение мобильного приложения в браузере. Django Templates используем **только** для админки и лендинга. |
| **Чаты (Realtime)** | **Django Channels** (WebSockets) | Чаты должны работать мгновенно, без перезагрузки страницы. |
| **Поиск** | **Elasticsearch** (опционально, на этапе роста) | На старте можно обойтись `SearchVector` в PostgreSQL, но при >500 СТО нужен Elastic. |
| **Очереди/Фон** | **Celery Beat** | Для автоматического списания абонентской платы и отправки напоминаний о записи. |
| **Контейнеризация** | **Docker** + **Docker Compose** | **Обязательно**. Чтобы на проде всё не упало из-за версии библиотеки. |

**Локальный запуск в текущем MVP-репозитории:** в корне проекта выполнить `docker compose up --build` — поднимаются `web` (Django, Gunicorn, `config.settings.docker`) и `db` (PostgreSQL 16). Перед стартом приложения выполняются миграции и сбор статики (`docker/entrypoint.py`, без shell-скрипта — совместимо с Windows). Сайт: http://127.0.0.1:8000/ , админка: `/secure-admin/`. Создание суперпользователя и переменные окружения — в комментариях к `docker-compose.yml` и в `.env.example`.

#### 2.2. Архитектура базы данных (Упрощенная ER-диаграмма в тексте)

Только ключевые сущности для атомарного понимания разработчиком.

**Пользователи (Custom User Model):**

**Профиль СТО (ServiceStation):**
- `id`, `owner (FK -> User)`, `name`, `legal_name` (юр. название), `inn` (опционально, для чеков).
- `address` (текст), `coordinates` (Point, PostGIS).
- `photos` (JSONField или отдельная модель).
- `subscription_plan` (free, basic_5000, pro_8000).
- `subscription_expires_at` (DateTime).

**Рабочие посты (WorkBay):**
- `id`, `sto (FK)`, `name` ("Подъемник №1", "Яма правая").
- `is_active`.

**Слоты записи (AppointmentSlot):**
- `id`, `bay (FK)`, `date`, `start_time`, `end_time`.
- `status` (free, booked, blocked_by_admin).
- **Важно:** Генерируются автоматически по шаблону работы СТО (Celery Beat задача раз в сутки генерит слоты на неделю вперед).

**Заказы/Записи (Order):**
- `id`, `client (FK -> User)`, `sto (FK)`, `slot (FK)`.

- `status` (pending_confirmation, confirmed, in_progress, completed, canceled, dispute).
- `rating` (от 1 до 5, может быть Null).
- `review_text` (может быть пустым, если оценка без текста).
- **Ограничение:** `review_text` может быть заполнен **ТОЛЬКО** если `status = 'completed'`.

**Платежи (Transaction):**
- `id`, `sto (FK)`, `amount`, `period_start`, `period_end`.
- `payment_method` (card, invoice), `status`.

**Чат (ChatRoom и Message):**

- **Правило безопасности:** Создать комнату может только Клиент (после записи) или СТО (после подтверждения записи).

#### 2.3. Функциональные требования (По модулям)

Разработчик должен видеть это как список Use Cases.

**Модуль 1: Аутентификация и Безопасность**
1. Регистрация **только по номеру телефона**. Подтверждение через 4-значный код (SMS-шлюз типа SmsAero/SMS.ru). Email опционален, но нужен для чеков.
2. Защита от брутфорса: Django Axes (блокировка по IP на 15 мин после 5 неверных попыток).
3. Капча (Google reCAPTCHA v3) на формах регистрации и входа.
4. Ролевая модель доступа:
   - **Клиент:** Видит список СТО, запись, свой профиль, чаты.
   - **Владелец СТО:** Видит список заказов, календарь слотов, финансовые отчеты (простые), может управлять подпиской.
   - **Мастер (сотрудник):** Видит только назначенные на него заказы (если внедряем, но в MVP можно обойтись ролью Владельца).
5. **GDPR/152-ФЗ:** Кнопка «Удалить аккаунт». При нажатии — анонимизация данных (номер телефона заменяется на `deleted_<uuid>`, ФИО стирается, история заказов остается для отчетности СТО, но без привязки к личности).

**Модуль 2: Каталог и Поиск СТО**
1. Поиск по названию, адресу, услугам (Full-text search в Postgres).
2. Геопоиск: «Показать все СТО в радиусе 5 км». Расчет расстояния через PostGIS.
3. Фильтры: «Есть свободные слоты сегодня», «Рейтинг > 4», «Тип услуг: Кузовной ремонт».
4. Карточка СТО:
   - Фотогалерея (сжатие изображений через `django-imagekit` при загрузке).
   - Средний рейтинг (звезды).
   - Ближайшие свободные слоты (лента времени).
   - **Отзывы:** Сортировка по дате (свежие сверху). Скрытие отзывов старше 6 месяцев в «архив», если их много.

**Модуль 3: Запись и Управление заказами**
1. **Выбор слота:** Клиент выбирает пост, время, вводит описание проблемы и госномер. **Важно:** Слот **сразу** блокируется на 15 минут (статус `temp_hold`). Если клиент не завершил запись за 15 минут — слот освобождается автоматически (Celery задача).
2. **Подтверждение СТО:** После записи заказ падает в ЛК СТО со статусом `pending_confirmation`. Владелец СТО должен нажать «Подтвердить». Если он этого не делает за 1 час — заказ отменяется, клиенту уходит пуш/смс «СТО не подтвердило, выберите другое время». Это фильтр «мертвых душ».
3. **Статусная модель:**
   - `pending` -> `confirmed` -> `in_progress` -> `completed`.
   - Перевод в `completed` **может сделать только СТО**.
4. **Отзыв:** После перехода в `completed` клиенту приходит Push/Email «Оцените визит». Отзыв можно оставить **один раз** в течение 7 дней. Редактирование запрещено.

**Модуль 4: Рейтинг и Репутация (Защита от накруток)**
1. **Право на отзыв:** Оставляют только `confirmed` пользователи с завершенным заказом.
2. **Система верификации отзыва:**
   - Если СТО жалуется на отзыв (кнопка «Пожаловаться на недостоверность»), отзыв не удаляется, но получает плашку **«На рассмотрении арбитража»**.
   - **Администратор платформы (ты)** видит скриншот диалога из чата и историю заказа. Только ты принимаешь решение: отклонить жалобу или скрыть отзыв с причиной.
3. **Анализ аномалий:** Скрипт (Celery задача раз в сутки), проверяющий новые СТО, у которых за 1 день появилось >5 оценок 5 звезд от новых пользователей. Высылает уведомление админу.

**Модуль 5: Платежи и Подписка (Billing)**
1. **Интеграция с эквайрингом:** Тинькофф Касса или ЮKassa. **Только рекуррентные платежи** (автосписание).
2. **Логика:**
   - СТО привязывает карту.
   - Система создает первый платеж на 1 рубль (верификация карты).
   - Каждое 1-е число месяца в 03:00 ночи Celery Beat запускает списание суммы согласно тарифу.
   - При неудачном списании — 3 попытки в течение 3 дней. Потом — блокировка кабинета (нельзя принимать новые заявки, профиль скрыт из выдачи).
3. **Фискализация:** Отправка чеков в ОФД через API ЮKassa. В назначении платежа указывать «Абонентская плата по тарифу "Базовый" за *Месяц*».

**Модуль 6: Чаты (Django Channels)**
1. **Безопасность:**
   - Нельзя написать в чат СТО, пока нет активного заказа.
   - Нельзя прикрепить файл `.exe`, только `.jpg, .png, .pdf`.
   - Модерация мата: подключение библиотеки `better_profanity` для фильтрации сообщений (замена звездочками).
2. **Уведомления:** При новом сообщении — **Web Push Notification** (Service Workers на фронте), дублирование на Email.

**Модуль 7: Административная панель (Django Admin кастомизация)**
1. Кастомные действия: «Массовая рассылка Push-уведомлений по выбранным СТО».
2. Дашборд: График новых регистраций СТО, кол-во записей за день, конверсия в отзыв.
3. Журнал аудита: Модель `AuditLog`, куда пишутся **все** действия администратора (кто и когда изменил статус заказа, удалил отзыв). Через `django-simple-history`.

#### 2.5. Производительность и Масштабирование (Что нужно предусмотреть в коде сейчас)

Ты сказал: «Чтобы выдерживал большие нагрузки». Внедри это в структуру кода **сразу**, чтобы не переписывать потом.
