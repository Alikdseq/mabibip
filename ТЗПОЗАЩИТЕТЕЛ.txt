Полное ТЗ: Комплексная защита контактов и антифрод в проекте «МаБибип»
Общая цель
Исключить утечку реальных телефонных номеров и другую нежелательную активность без использования платных внешних сервисов. Все механизмы реализуются на стороне сервера и клиента (браузер) в рамках текущего стека: Django 4.x, HTMX, vanilla JS.

1. Маскировка номера телефона на клиенте
1.2 Маска номера с ручным раскрытием
Функциональность:

В карточке объявления (и в списке) номер отображается в виде +7 (XXX) XXX-XX-XX.

Рядом – кнопка «Показать номер».

При клике – всплывает предупреждающее модальное окно: «Осторожно! Не переводите предоплату и не сообщайте личные данные. Продолжить?»

После подтверждения номер полностью показывается и остаётся видимым 5 минут (отсчёт на клиенте). По истечении 5 минут номер снова маскируется, кнопка сбрасывается.

Для неавторизованных – требование войти (редирект на логин).

Техническая реализация:

На сервере: API GET /api/listings/{id}/reveal-phone/.

Проверка авторизации.

Проверка лимитов (см. раздел 3.1).

Возвращает настоящий номер и время начала показа (для синхронизации).

Клиент: HTMX-запрос или fetch(). После получения – обновление соответствующего DOM-элемента.

Хранить состояние показа в localStorage или в замыкании JS (на странице). Таймер на 5 минут с обратным отсчётом; по истечении – скрыть номер через замену текста.

Защита от парсинга: боты не могут нажать кнопку (reCAPTCHA не требуется, так как уже авторизованы, и срабатывает лимит запросов). Серверный рендеринг телефонной маски не содержит настоящих цифр.

Модель:
Создать PhoneRevealLog:

user (ForeignKey), listing (ForeignKey), revealed_at (DateTimeField).
Используется для лимитов и аудита.

1.3 Запрет копирования и выделения номера
После раскрытия номера, контейнер с номером получает CSS-свойства:

css
-webkit-user-select: none;
user-select: none;
pointer-events: none; /* дополнительно предотвращает выделение */
Дополнительно номер рендерится как <span> без пробелов, с возможным разбиением на части (например, +7<span>999</span><span>123</span>...). Это усложняет автоматическое копирование.

Контекстное меню (правая кнопка) на этом элементе запрещено через oncontextmenu="return false;".

Серверный рендер номера в виде картинки – избыточен и неудобен для пользователя (плохая доступность). Вместо этого придерживаемся CSS+JS подхода.

Для мобильных устройств:
Дополнительно блокируем долгое нажатие (тап-холд) через addEventListener('touchstart', e => e.preventDefault()) только на этом элементе (с осторожностью, чтобы не сломать скролл).

1.4 Динамическая маскировка для каждого пользователя (прокси-ссылка)
Вместо отображения номера показывается кнопка «Позвонить» (или «Связаться»). При нажатии:

Отправляется запрос на сервер: POST /api/calls/request/ (детали звонков – отдельное ТЗ).

Сервер возвращает уникальный идентификатор сессии или сразу перенаправляет на интерфейс звонка внутри приложения (WebRTC).

Сам номер не появляется в DOM вообще.

Реализовать постепенно: как первый шаг – заменить показ номера на кнопку «Связаться через приложение», которая открывает чат или звонок. Номер при этом не отображается, даже после авторизации. (Если прокси-звонки уже работают, используем их).

2. Подтверждение личности через соцсети (OAuth)
Цель: Усложнить массовую регистрацию ботов.

Реализация:

Добавить в систему аутентификации возможность входа через:

Google (OAuth 2.0),

Apple ID (Sign in with Apple),

ВКонтакте (OAuth 2.0).

Использовать библиотеку social-auth-app-django (или django-allauth).
Рекомендация: django-allauth – активно поддерживается, имеет встроенные провайдеры для Google, Apple, VK.

Настроить соответствующие приложения в консолях Google Cloud, Apple Developer, VK.

Проверка возраста аккаунта: при логине через Google – можно запросить https://www.googleapis.com/oauth2/v1/userinfo?alt=json и проверить дату создания (если доступно).
Для ВКонтакте – метод users.get с полем ?fields=... может дать дату регистрации?
Если не доступно, можно использовать эвристику: дату первого входа через OAuth в наше приложение (время создания SocialAccount). Требовать, чтобы аккаунт был создан больше 6 месяцев назад. Если аккаунт новее — запрещать использование некоторых функций (публикация объявлений) до подтверждения телефона или email.

Модель: минимально – стандартная модель SocialAccount из allauth. Дополнительные поля не нужны.

3. Лимиты на показ номеров
Функциональность:

Один пользователь за 1 час может раскрыть не более 5 разных номеров.

За 24 часа – не более 20.

При превышении – временная блокировка просмотра контактов на 24 часа (всех объявлений). Пользователь видит сообщение «Вы превысили лимит раскрытия номеров. Попробуйте позже».

Реализация:

Использовать модель PhoneRevealLog (п. 1.2).

В эндпоинте /reveal-phone/ выполнить запрос:

python
recent_hour = PhoneRevealLog.objects.filter(user=request.user, revealed_at__gte=now()-timedelta(hours=1)).count()
recent_day = PhoneRevealLog.objects.filter(user=request.user, revealed_at__gte=now()-timedelta(days=1)).count()
if recent_hour >= 5 or recent_day >= 20:
    return Response({'error': 'лимит исчерпан'}, status=429)
При превышении сохранить блокировку в профиле пользователя: добавить поле contact_view_blocked_until (DateTimeField, null). При превышении устанавливать now() + 24h. При следующем запросе проверять: если время ещё не прошло – отказ.

Очищать блокировку автоматически при следующей успешной проверке времени.

Оптимизация: можно использовать Redis для быстрых счётчиков, но на начальном этапе БД достаточно (индексы на user, revealed_at).

4. Антифрод и модерация
4.1 Кнопка «Пожаловаться» с автоматической блокировкой
Добавить в каждый элемент объявления кнопку «Пожаловаться» (флажок или иконка). Видна только авторизованным.

При нажатии – HTMX-запрос POST /api/listings/{id}/report/.
Создаётся запись ListingReport:

listing, reported_by, reason (текст, опционально), created_at.

При создании проверяется количество уникальных жалоб от разных пользователей за последние N дней (например, 7 дней). Если жалоб >= 3 и статус объявления «активно» → автоматически перевести в статус «скрыто» (is_visible=False или статус reported).

Создать задачу для администратора: ежедневный просмотр скрытых объявлений в админке, возможность восстановить.

Модель: ListingReport. Важно учитывать, чтобы нельзя было накрутить жалобы с одного аккаунта: группировка по reported_by.

4.2 Анализ поведения (антифрод-правила)
Реализовать сервис AntiFraudService с методами, вызываемыми из сигналов или периодической задачи:

Частая смена номера телефона:

При сохранении профиля пользователя, проверяем логи изменений номера (модель PhoneChangeLog).
Если за последние 24 часа было >2 изменений – заморозить аккаунт: установить is_active=False или добавить в «подозрительные» (флаг is_suspicious=True), запретить публикацию объявлений, отправить сигнал администратору.

Одинаковые фотографии в разных аккаунтах:

При загрузке фото (для объявления или профиля) вычислять perceptual hash (phash). Хранить в модели ImageHash (одно фото может использоваться в нескольких объявлениях? Лучше связывать с загруженным файлом).

При добавлении нового фото проверять, не встречается ли такой же phash у другого активного пользователя. Если да – пометить обоих пользователей подозрительными.

Можно использовать библиотеку imagehash (PIL) и расстояние Хемминга <= 5 считать дубликатом.

Список разных номеров в тексте объявления:

Регулярное выражение для поиска всех телефонных номеров в тексте (и description). Если количество различных номеров >1 (или есть признаки «контакты вне сервиса»), помечать объявление на модерацию, автоматически заменять номера на плейсхолдеры с предупреждением.

Реализация: сигналы pre_save для User (телефон), post_save для ClassifiedAd (текст).

4.3 Скрытие номера после завершения сделки
При изменении статуса объявления на «продано» (или «снято с публикации») должны быть отозваны все ранее показанные номера:

Если номер был раскрыт по маске (п.1.2) – очистить кэш отображения (не актуально, т.к. показ динамический). Но главное – дальнейшие запросы на раскрытие номера для этого объявления должны возвращать ошибку «Объявление неактуально».

Реализовать проверку статуса объявления в эндпоинте reveal-phone: если статус is_sold или is_deleted – возвращать 404/403.

Дополнительно: если в объявлении был прямой номер (в старых версиях) – заменить текст на «Продано» модератором.

4.4 Модерация объявлений перед публикацией
В процессе создания/редактирования объявления (вьюха или форма) вызывать функцию validate_listing_text(text).

Алгоритм:

Поиск номеров телефонов (российские, международные) через regex.

Поиск ссылок на мессенджеры (t.me, wa.me, vk.com, ...).

Поиск email.

Если найдены – действие:

Для новых пользователей (зарегистрирован меньше X дней): отклонять публикацию с сообщением «Запрещено размещать контактные данные. Используйте встроенный чат.»

Для остальных – объявление сохраняется, но устанавливается moderation_status = 'pending' и флаг is_visible = False до проверки модератором.

Бесплатно: регулярные выражения, реализованные на Python.

Модель: добавить поле moderation_status в ClassifiedAd: ('auto_approved', 'pending', 'approved', 'rejected').

5. Обучение пользователей
5.1 Экран безопасности при первом контакте
При первой попытке раскрыть номер или написать в чат – показывать модальное окно с предупреждением (только один раз).

Использовать localStorage или cookie: ключ safety_warning_seen. Если нет – показать модалку, после закрытия записать в localStorage.

Модальное окно содержит:

«Никому не переводите предоплату»

«Общайтесь только внутри приложения»

«Сообщайте о подозрительных пользователях»

Реализация: чистый JS + HTML/CSS, без дополнительных библиотек.

Показывать всегда, вне зависимости от статуса авторизации.

7. Безопасность и конфиденциальность
Все эндпоинты раскрытия и подачи жалоб должны быть защищены CSRF (для сессий) и авторизацией.

Необходимо журналировать все раскрытия номеров для аудита.

Проверки anti-fraud не должны замедлять основные действия пользователя – хэширование изображений выполнять асинхронно (через task queue или on_commit).

При показе предупреждения не должно быть возможности его закрыть нажатием вне модального окна, чтобы гарантировать прочтение (но только первый раз).

8. Ожидаемые результаты (Definition of Done)
В карточке объявления номер замаскирован, раскрывается по кнопке на 5 минут.

Копирование номера стандартными средствами невозможно.

Авторизованный пользователь не может запросить больше 5 номеров в час.

OAuth-логин работает для Google, Apple, VK (можно реализовать хотя бы Google для MVP).

При трёх жалобах от разных пользователей объявление автоматически скрывается.

В админке видны заблокированные объявления и подозрительные аккаунты.

Запрещено публиковать прямые номера в тексте объявления (модерация).

После отметки «продано» номер недоступен для запроса.

Первый раз при контакте показывается обучающая модалка.

В каждом объявлении присутствует плашка безопасности.

9. Примерные файлы и код
listings/views.py (фрагмент):

python
from django.utils import timezone
from datetime import timedelta
from .models import PhoneRevealLog

@login_required
def reveal_phone(request, listing_id):
    listing = get_object_or_404(Listing, pk=listing_id, is_visible=True, moderation_status='approved')
    if listing.is_sold:
        raise Http404
    # Check block
    if request.user.contact_view_blocked_until and request.user.contact_view_blocked_until > timezone.now():
        return JsonResponse({'error': 'Вы временно заблокированы'}, status=429)
    # Check limits
    hour_ago = timezone.now() - timedelta(hours=1)
    day_ago = timezone.now() - timedelta(days=1)
    if PhoneRevealLog.objects.filter(user=request.user, revealed_at__gte=hour_ago).count() >= 5 or \
       PhoneRevealLog.objects.filter(user=request.user, revealed_at__gte=day_ago).count() >= 20:
        request.user.contact_view_blocked_until = timezone.now() + timedelta(days=1)
        request.user.save()
        return JsonResponse({'error': 'Лимит исчерпан'}, status=429)
    # Log and return number
    PhoneRevealLog.objects.create(user=request.user, listing=listing)
    return JsonResponse({'phone': listing.owner.phone.as_e164, 'revealed_at': timezone.now().isoformat()})
