# Атомарный план разработки полного веб-приложения «ПроМастер» по полному ТЗ

**Назначение:** пошаговый профессиональный чеклист от **уже реализованного MVP** (см. `docs/PLAN-MVP-ATOMIC.md`) до соответствия **полному** техническому заданию в `ТЗ.txt` и требованиям **Enterprise-ready** архитектуры. Каждый пункт — одно логически завершённое действие с указанием артефакта, **обязательных тестов**, контроля **безопасности** (синхронизировано с `docs/07-kompleksnaya-zaschita-dannyh-i-protivodejstvie-atakam.md`) и учёта **Docker**-окружения.

**Принцип:** не дублировать выполненную работу MVP; явно помечать «выполнено» и строить дальнейшие фазы как **дельту** к полному ТЗ.

**Связанные документы:**

| Документ | Роль |
| :-- | :-- |
| `ТЗ.txt` | Полное ТЗ (модули 1–7, безопасность, производительность, юридический фундамент) |
| `docs/PLAN-MVP-ATOMIC.md` | MVP: фазы 0–10 — **базовая линия «сделано»** |
| `docs/07-kompleksnaya-zaschita-dannyh-i-protivodejstvie-atakam.md` | Угрозы, чеклисты, нормативка152-ФЗ, матрица «фича → угрозы» |
| `docker-compose.yml`, `Dockerfile`, `docker/` | Контейнеризация: все новые сервисы (Celery, Beat, Redis, ASGI) должны воспроизводиться через `docker compose` |

## 0. Договорённости по качеству (обязательны для всех фаз после MVP)

### 0.1. Тестирование (каждая фича — с тестами)

**Шаг 0.1.1.** Любой новый публичный/приватный эндпоинт, Celery-задача с бизнес-логикой, consumer WebSocket — сопровождаются минимум:

- **unit/integration** тестами (`pytest`, `pytest-django`);
- для критичных путей (оплата, смена статуса заказа, доступ к чату) — **негативными** сценариями (чужой ресурс, неверная роль, повтор webhook).

**Шаг 0.1.2.** Регрессия MVP: после каждой крупной фазы — полный прогон `pytest`; целевое покрытие для новых модулей **≥ 75%** (критичные — **≥ 85%**): `billing`, `messaging` (чаты), `subscriptions` (рекуррент), геопоиск, модерация отзывов.

**Шаг 0.1.3.** Выделить маркеры: `@pytest.mark.slow` (e2e-flow), `@pytest.mark.security` (IDOR, rate limit smoke), `@pytest.mark.celery` (задачи с eager-режимом).

**Шаг 0.1.4.** Перед релизом: статический анализ (`ruff`), скан зависимостей (`pip-audit` или аналог), по документу `07` — опционально `bandit` на CI.

### 0.2. Безопасность (синхронизация с документом `07`)

Перед началом **каждого** спринта/фазы команда проходит **часть E.1** файла `07-kompleksnaya-zaschita-...md` (данные, эндпоинты, атомарность, внешние вызовы).

Перед **merge** — **часть E.2** (IDOR, сырой SQL, XSS, файлы, секреты, логи).

Перед **прод-релизом** — **часть E.3** (TLS, бэкапы, Sentry с маскированием, админка+МФА).

### 0.3. Docker и окружения

**Шаг 0.3.1.** Целевая локальная схема: `web` (Django), `db` (PostgreSQL + PostGIS), `redis`, `celery-worker`, `celery-beat`, `asgi` (Uvicorn + Channels), при необходимости `nginx` (или nginx на хосте на проде).

**Шаг 0.3.2.** Все новые зависимости — в `requirements/*.txt`; переменные — в `.env.example` без секретов.

**Шаг 0.3.3.** Отдельный `config/settings/docker.py` (или расширение существующего) должен включать брокер Celery, Redis cache/session, `CHANNEL_LAYERS`.

### 0.4. Шифрование и ПДн

**Шаг 0.4.1.** Персональные и платёжные идентификаторы не логировать в открытом виде; в Sentry — scrubbing чувствительных полей (см. `07`, часть B.11).

**Шаг 0.4.2.** Бэкапы БД: шифрование перед выгрузкой в объектное хранилище (отдельная фаза инфраструктуры); ключи — только из secret manager / env на сервере бэкапа.

## 3. Фаза F0 — Юридический фундамент (6 документов + продуктовые галочки)

### Цель

Соответствие части 1 `ТЗ.txt` и 152-ФЗ/54-ФЗ на уровне **процесса и UI** (контент документов готовит юрист; разработка — версии, маршруты, версионирование принятий).

### Атомарные шаги

**Шаг F0.1.1.** Создать в репозитории каталог `docs/legal/` (или `templates/legal/`) для **Markdown-исходников** шести документов из таблицы ТЗ (политика конфиденциальности, пользовательское соглашение, оферта СТО, согласие на ПДн, политика ИБ, оферта платных услуг).

**Шаг F0.1.2.** Модель `LegalDocumentVersion` (slug, title, effective_at, content_key или путь к файлу, checksum).

**Шаг F0.1.3.** Модель `UserConsent`: FK `user`, тип согласия (enum), `document_version`, `accepted_at`, `ip` (опционально, обсудить с юристом), `user_agent` (укороченный).

**Шаг F0.1.4.** При регистрации / первом входе в ЛК СТО — **обязательные** чекбоксы с ссылками на актуальные версии; без галочек — `400` и понятная ошибка на API.

**Шаг F0.1.5.** Эндпоинт или страница «Архив версий документов» для прозрачности.

**Тест F0.T1.** Регистрация без согласия — отказ.  
**Тест F0.T2.** Согласие сохраняется с корректным FK на версию.  
**Безопасность (`07`):** не хранить лишние ПДн в consent без правовой необходимости; маскировать IP в админ-списках при экспорте.

**Docker:** не требует отдельного сервиса; миграции в `web`.

## 4. Фаза F1 — Аутентификация и защита входа (SMS, капча, axes, право на забвение)

### Дельта к MVP

MVP: email + пароль. ТЗ: **телефон как логин**, SMS OTP, reCAPTCHA v3, django-axes, удаление аккаунта с анонимизацией.

### Атомарные шаги

**Шаг F1.1.1.** Выбрать SMS-провайдера (SmsAero, SMS.ru и т.д.); обёртка `apps/users/sms.py` с интерфейсом `send_otp(phone, code)` — **никаких** секретов в коде.

**Шаг F1.1.2.** Модель/таблица `PhoneVerificationChallenge`: phone (нормализованный E.164), код hash (не хранить plaintext), attempts, locked_until, created_at, ip.

**Шаг F1.1.3.** Эндпоинты: запрос кода (rate limit по phone+IP), проверка кода, завершение регистрации; **блокировка** после N попыток (см. `07` B.2 перебор SMS).

**Шаг F1.1.4.** Интеграция **Google reCAPTCHA v3** на серверной валидации для register/login (score threshold из env).

**Шаг F1.1.5.** Подключить **django-axes**; настройки блокировки согласовать с ТЗ (например 5 неудач / 15 мин); тестировать за прокси (`SECURE_PROXY_SSL_HEADER`).

**Шаг F1.1.6.** Миграция пользователей: поле `phone` уникальное, `is_phone_verified`; стратегия сосуществования с email (email опционален для чеков — ТЗ).

**Шаг F1.1.7.** Фича **«Удалить аккаунт»**: сервис анонимизации — телефон → `deleted_<uuid>`, ФИО очистка, заказы сохраняются для СТО без связи с личностью (уточнить с юристом схему ключей).

**Тест F1.T1.** Перебор OTP — lockout.  
**Тест F1.T2.** Axes блокирует IP/username после порога.  
**Тест F1.T3.** Удаление аккаунта: пользователь не может логиниться; заказы остаются.  
**Тест F1.T4.** IDOR: нельзя запросить чужую верификацию.

**Безопасность (`07`):** B.2 (OTP), B.7 (throttling DRF на auth), логи без кода.

**Docker:** переменные SMS/recaptcha/axes в `.env.example`.

## 6. Фаза F3 — Генерация слотов (Celery Beat) и Redis hold 15 минут

### Дельта к MVP

MVP: слоты вручную/импорт; автоотмена pending. ТЗ: автогенерация на неделю вперёд по шаблону; hold слота **15 мин** в Redis; Celery освобождает.

### Атомарные шаги

**Шаг F3.1.1.** Модель `WorkingHours`: FK станция или пост, день недели, `opens_at`, `closes_at`, шаг слота (например 30 мин), перерывы (JSON или отдельная модель).

**Шаг F3.1.2.** Задача Celery `generate_weekly_slots`: для каждой станции создаёт слоты на `today + 7` дней без дублирования (идемпотентность по `(bay, date, start_time)`).

**Шаг F3.1.3.** Celery Beat расписание: ежедневно (например 02:00 МСК) + ручной `manage.py` для догоняния.

**Шаг F3.1.4.** При начале записи клиентом: `SET slot_hold:{slot_id} {user_id} NX EX 900`; при успешном создании заказа — удалить ключ; фоновая задача чистит просроченные holds.

**Шаг F3.1.5.** Согласовать с `slot_is_bookable`: слот недоступен, если есть hold чужим пользователем или активная бронь.

**Тест F3.T1.** Генерация не плодит дубликаты при повторном запуске.  
**Тест F3.T2.** Hold блокирует чужую бронь; истечение — разблокирует.  
**Тест F3.T3.** Race: два клиента — один hold побеждает.

**Безопасность (`07`):** B.3 race, B.9 Redis — TTL, не хранить лишние ПДн в ключах.

**Docker:** сервисы `redis`, `celery-worker`, `celery-beat`; зависимость `web` от `redis`.
