# Security roadmap (ProMaster)

Дата: 2026-04-21  
Основание: `docs/security/threat-model.md`, кодовая база (Django/DRF/Channels), найденные факты и внесённые фиксы.

## 0) Уже исправлено в рамках текущей итерации
- **XSS‑класс риск**: убрано исполнение `<script>` из подгруженного HTML в `templates/stations/station_detail.html`.\n- **WS устойчивость**: `apps/chat/consumers.py` больше не падает при проблемах с Redis rate‑limit (возвращает контролируемую ошибку).\n- **Логи и OTP**: `apps/users/sms.py` больше не логирует OTP и полный телефон.\n- **Secrets hygiene**: удалён коммитнутый `.env` (должен храниться только локально/в секрет‑хранилище).\n- **Monitoring exposure**: порты monitoring‑стека в `docker-compose.yml` привязаны к `127.0.0.1`.\n- **Docker hardening**: `Dockerfile` переведён на non‑root `USER app`.\n- **Proxy correctness**: `SECURE_PROXY_SSL_HEADER/USE_X_FORWARDED_HOST` включены для prod, и добавлены opt‑in флаги для docker.\n- **Audit minimization**: `apps/audit/utils.py` редактирует payload, `AuditLog.__str__` больше не печатает телефон.\n\n## 1) 🔴 Критические (исправить немедленно)

### 1.1 Ротация секретов после утечки `.env`
- **Риск**: компрометация reCAPTCHA/SMTP/любых ключей, которые могли быть в `.env`.\n- **Действие**: ротировать все ключи, которые когда‑либо попадали в репозиторий/CI лог/артефакты.\n- **Артефакты**: добавить регламент “как ротировать” + список секретов (внутренний документ).

### 1.2 Контур Nginx: защита входных точек
- **Риск**: брутфорс логина, флад POST эндпоинтов и вебхуков.\n- **Действие**: на Nginx включить `limit_req` (login, register, chat send endpoints, webhook) + базовые security headers.\n- **Оценка**: 0.5–1 день.

## 2) 🟠 Высокие (ближайшая неделя)

### 2.1 CSP и отказ от inline‑скриптов (поэтапно)
- **Риск**: DOM XSS при появлении новых мест с `innerHTML`.\n- **Действие**: вынести inline JS в статические файлы, настроить CSP (с nonce или без unsafe-inline).\n- **Оценка**: 2–5 дней (зависит от числа шаблонов).

### 2.2 Стабилизация деградаций Redis/Channels
- **Риск**: при деградации Redis могут ломаться чаты/слоты/сессии.\n- **Действие**: единая стратегия fail‑open/fail‑closed по подсистемам + алерты.\n- **Оценка**: 1–2 дня.

### 2.3 Политика ретенции для чатов/аудита/логов
- **Риск**: избыточное хранение ПДн.\n- **Действие**: задать сроки хранения + периодические задачи очистки/архивации.\n- **Оценка**: 1–3 дня.

## 3) 🟡 Средние (ближайший месяц)

### 3.1 Тесты безопасности на IDOR (регресс)
- **Цель**: зафиксировать, что client/owner не могут видеть/менять чужие сущности.\n- **Действие**: e2e/интеграционные тесты для booking/chat/review.\n- **Оценка**: 2–4 дня.

### 3.2 Доработка audit governance
- **Цель**: гарантировать отсутствие ПДн в `AuditLog.payload` по правилам.\n- **Действие**: добавить allowlist событий/полей, и unit‑тесты на редактирование.\n- **Оценка**: 1–2 дня.

## 4) 🟢 Долгосрочно
- **WAF/CDN + DDoS** (Cloudflare/аналог)\n- **2FA для админов/ERP** (если включите по риск‑профилю)\n- **SAST/Dependency scanning** в CI (bandit/pip-audit/Dependabot)\n\n## 5) Чек‑лист повторного аудита (после внедрений)
- Секреты: нет секретов в git, ротация проведена.\n- Nginx: TLS, rate limits, security headers.\n- Django prod: secure cookies, proxy headers.\n- IDOR: тесты зелёные.\n- Чат: WS ACL + rate-limit работает, деградация Redis контролируется.\n- Логи/аудит: нет ПДн/OTP/токенов, ретенция определена.\n- Бэкапы: шифрование + регулярный restore‑drill.
