# 152‑ФЗ gap‑анализ (ProMaster)

Дата: 2026-04-21  
Основание: анализ репозитория `c:/promasterov` (Django/DRF/Channels), юридических документов в `docs/legal/*` и механизмов фиксации согласий (`apps/legal/*`).

## 1) Что уже соответствует (по коду/докам)

- **Политика конфиденциальности и согласие на ПДн**: есть тексты документов в `docs/legal/privacy.md`, `docs/legal/pd_consent.md` и модель версий документов + фиксации согласий (`apps/legal/models.py`).\n- **Фиксация согласий при регистрации**: регистрационные формы требуют чекбоксы (privacy/user_agreement/pd_consent), а `apps/users/views.py` записывает согласия через `apps/legal/services.py`.\n- **Право на удаление/обезличивание**: реализовано как анонимизация (с сохранением FK на заказы) — `apps/users/services_anonymize.py`, endpoint `users:account_delete`.\n- **Минимизация ПДн в Sentry**: включён `before_send` фильтр без default PII — `config/settings/base.py`.\n- **Шифрование бэкапов**: бэкапы БД шифруются `age` перед выгрузкой в S3 — `docker/backup/backup_to_s3.sh`.\n\n## 2) Частично выполнено / требует уточнения в инфраструктуре\n\n- **Шифрование “в покое” (at-rest) для живой БД/дисков**:\n  - В коде нет field-level encryption для чувствительных полей (телефон/адрес/VIN/госномер).\n  - Требование закрывается **на уровне инфраструктуры** (шифрование дисков/managed DB encryption/KMS). Нужна фиксация в эксплуатационных требованиях и регламенте.\n- **Доступ к логам/аудиту**:\n  - Аудит в БД есть (`apps/audit/*`), но нужно формализовать права доступа, ретенцию, и процедуру выдачи/отзыва доступа.\n\n## 3) Пробелы (что добавить, чтобы соответствие было “железным”)\n\n### 3.1 Организационные меры (документально)\n- **Модель угроз/классификация ИСПДн и уровни защищённости** (ПП РФ №1119, приказ ФСТЭК №21): оформить как отдельный комплект документов (или расширить вашу Политику ИБ `docs/05-politika-informacionnoj-bezopasnosti.md`).\n- **Регламент реагирования на инциденты**: сроки, роли, фиксация событий, порядок уведомлений.\n- **Порядок доступа к ПДн**: кто, по какой процедуре, журналирование, периодический пересмотр.\n\n### 3.2 Технические меры (в продукте/деплое)\n- **Секреты/ключи**: запрет хранения секретов в git; хранение в env/secret store; ротация при утечках.\n- **Сегментация сети**: DB/Redis только во внутренней сети, без публикации портов наружу.\n- **Ретенция и удаление**:\n  - Сейчас есть анонимизация аккаунта, но нужно формализовать сроки хранения разных сущностей (чаты/аудит/заказы) и механизм удаления/архивации по срокам.\n- **Маркетинговое согласие (если будут рассылки)**:\n  - В текущем виде есть согласие на ПДн и пользовательское соглашение; отдельного согласия на маркетинговые сообщения нет. Если функционал рассылок появится — нужно отдельное согласие + возможность отписки.\n\n## 4) Рекомендации (приоритеты)\n\n- **Сразу**: секреты из репозитория, ротация ключей, ограничение доступа к мониторингу/логам.\n- **Ближайший спринт**: закрепить требования к encryption-at-rest в инфраструктуре; ретенция логов/аудита; минимизация ПДн в логах.\n- **Дальше**: классификация ИСПДн, комплект организационных документов, регулярный внутренний аудит.\n+
