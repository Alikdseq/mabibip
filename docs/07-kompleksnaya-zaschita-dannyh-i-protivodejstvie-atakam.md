## Часть A. Область действия и активы

### A.1. Защищаемые активы

| Актив | Критичность | Примечание |
| :-- | :-- | :-- |
| Персональные данные (телефон, e-mail, данные об авто, переписка) | **Высокая** | 152-ФЗ |
| Учётные данные (пароли, токены, сессии, JWT) | **Высокая** | Компрометация = захват аккаунтов |
| Платёжные идентификаторы, история транзакций | **Высокая** | Через ЮKassa и др.; токены, не PAN на своих серверах |
| Бизнес-логика (слоты, заказы, подписки СТО) | **Высокая** | Финансовые и репутационные потери |
| Секреты (SECRET_KEY, DB, Redis, API ключи SMS/карт) | **Критическая** | Утечка = полный компромисс |
| Резервные копии БД | **Высокая** | Шифрование, изолированное хранение |
| Логи и метрики | **Средняя** | Риск утечки ПДн в трассах — маскирование |
| Исходный код и CI/CD | **Высокая** | Supply-chain атаки |

### A.2. Граница системы (поверхность атаки)

- Публичный **веб/API** (Django + DRF).
- **WebSocket** (Django Channels) — чаты.
- **Админ-панель** (скрытый URL + МФА).
- **Фоновые воркеры** Celery (нет прямого выхода в интернет без необходимости).
- **PostgreSQL, Redis** — только внутренняя сеть.
- **Интеграции:** SMS, e-mail, ЮKassa, геокодирование (при наличии).
- **Фронтенд** (Vue) — XSS через DOM, небезопасное хранение токенов.

## Часть B. Классификация угроз и атак (что учитываем обязательно)

Ниже — **не исчерпывающий**, но **практический** перечень для B2C/B2B веб-сервиса. При проектировании каждой фичи сверяйтесь: «какая строка таблицы затрагивается?».

### B.1. Инъекции и манипуляция запросами

| Угроза | Суть | Меры (обязательные для «ПроМастер») |
| :-- | :-- | :-- |
| **SQL-инъекция** | Подмена параметров запроса к БД | Только **Django ORM** / параметризованные запросы; **запрет** `raw()` без security review; минимальные права пользователя БД |
| **ORM-логические ошибки** | `filter()` по пользовательским полям без ограничения | Всегда **ограничивать** queryset текущим пользователем / ролью (**IDOR**, см. ниже) |
| **Command injection** | Вызов shell из Python | **Не** использовать `os.system`, `subprocess` с пользовательским вводом; если неизбежно — жёсткий allow-list |
| **SSRF** | Сервер запрашивает URL, заданный атакующим | Валидация URL, allow-list хостов для исходящих запросов (геокодер, webhooks); таймауты |
| **NoSQL/LDAP** | При появлении иных хранилищ | Параметризация, отсутствие конкатенации в запросах |

### B.2. Аутентификация, сессии, токены

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Перебор паролей / OTP** | Массовые попытки входа | **django-axes** (или аналог), **rate limit** на `/api/auth/*`, блокировка после N неудач; **капча** (reCAPTCHA v3) на регистрации/логине по ТЗ |
| **Перебор SMS-кодов** | Угадывание 4-значного кода | Лимит попыток **на номер и на IP**, увеличивающаяся задержка, при необходимости блокировка канала; логирование аномалий |
| **Фиксация сессии** | Подмена session id | Регенерация сессии после логина; `SameSite` cookie |
| **Кража JWT** | XSS / перехват | Короткий TTL access-token; refresh с ротацией; **HttpOnly** где применимо; хранение на фронте — осознанный риск (предпочтение memory > localStorage для access) |
| **Слабые пароли** | У пользователей СТО/админа | Политика длины/сложности для ролей с расширенными правами; **МФА для admin** |
| **Утечка SECRET_KEY** | Подделка сессий Django | Хранение в **секретах** окружения, ротация при компрометации |

### B.3. Авторизация и бизнес-логика

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **IDOR** | Доступ к чужому заказу/чату по id | Проверка **владения** ресурсом на **каждом** эндпоинте; не полагаться только на «секретность» UUID |
| **Privilege escalation** | Клиент вызывает эндпоинт админа | Разделение URL/API; **явные** permission classes DRF; тесты на роли |
| **Race condition (слоты)** | Двойное бронирование одного слота | **Redis SET key NX EX** (как в ТЗ), транзакции БД с `select_for_update` где уместно; идемпотентность создания заказа |
| **Обход статусов заказа** | Клиент сам ставит `completed` | Переходы статусов только **серверной** машиной состояний; whitelist переходов по роли |
| **Повторный отзыв** | Два отзыва на один заказ | Уникальность `(order_id)` для отзыва; проверка на уровне БД |

### B.4. XSS, CSRF, клиентская безопасность

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Stored/Reflected XSS** | Скрипт в отзыве/чате | Экранирование в API для HTML; на Vue — **не** использовать `v-html` для пользовательского контента без санитизации (**DOMPurify**); CSP заголовки (см. Nginx) |
| **CSRF** | Действие от имени пользователя | Session-cookie: **CSRF middleware**; API JWT: не полагаться на cookie для state-changing без доп. защиты |
| **Clickjacking** | Встраивание сайта в iframe | `X-Frame-Options: DENY` или `SAMEORIGIN`; `Content-Security-Policy: frame-ancestors 'none'` |
| **Open redirect** | `next=` на вредоносный сайт | Allow-list доменов для редиректов |

### B.5. Загрузка файлов и вложения

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Вредоносные файлы** | .exe, двойные расширения | Allow-list **только** `.jpg, .png, .pdf` (по ТЗ); проверка **MIME** и **magic bytes**, не только расширения |
| **Path traversal** | `../../etc/passwd` | Сохранение под **сгенерированными** именами; без пользовательских путей |
| **DoS через размер** | Огромные файлы | Лимит размера на Nginx и в приложении |

### B.6. WebSocket (Django Channels)

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Доступ к чужой комнате** | Подбор channel / room id | Аутентификация в **consumer**; проверка участия в **order** перед join |
| **Flood** | Спам сообщениями | Rate limit **на пользователя** в consumer или через Redis-счётчики |
| **Отсутствие авторизации** | Анонимный socket | Только после валидного JWT / session |

### B.7. API-специфика (DRF)

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Mass assignment** | Передача лишних полей в PATCH | Явные `serializer.fields`; **read_only** для чувствительных полей |
| **Чремерное раскрытие данных** | `select_related` забыли, но отдали лишнее | Отдельные **serializer** для списка и деталки; не отдавать внутренние id партнёров без нужды |
| **BOLA (OWASP API #1)** | = IDOR для API | Тесты + code review чеклист |
| **Отсутствие лимитов** | Скрейпинг каталога | **Throttling** DRF; лимиты для анонимов/юзеров по ТЗ |

### B.8. Инфраструктура и сеть

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **DDoS / L7 flood** | Затопление запросами | **Nginx** `limit_req`, `limit_conn`; при росте — CDN/WAF (Cloudflare, QRATOR и т.д.) |
| **Небезопасный TLS** | Слабые шифры | Современные cipher suites; отключение старых протоколов |
| **MITM** | Перехват трафика | **HTTPS везде**; **HSTS** |
| **Сканирование админки** | Перебор `/admin` | **Нестандартный путь** + IP allow-list (опционально) + **МФА** |
| **Открытые порты БД** | Postgres/Redis в интернете | Только **private network**; security groups |

### B.9. Redis и Celery

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Команды без пароля** | FLUSHALL извне | **requirepass** / ACL Redis; bind на внутренний интерфейс |
| **Pickle в Celery** | RCE через сообщения | **JSON serializer** для задач; не передавать непроверенные объекты |
| **Утечка задач** | Чувствительные данные в брокере | Минимизировать payload; TTL для ключей hold слотов |

### B.10. Зависимости и цепочка поставки

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **Уязвимые пакеты** | CVE в Django/DRF | **Dependabot**/Renovate; регулярный `pip-audit` / `safety` |
| **Компрометация CI** | Утечка секретов из GitHub Actions | **GitHub Environments**, OIDC, запрет секретов в логах |
| **Подделка образов** | Docker | Базовые образы из доверенных репозиториев; pin digest |

### B.11. Логи, мониторинг, утечки через ошибки

| Угроза | Суть | Меры |
| :-- | :-- | :-- |
| **ПДн в traceback** | Телефон в Sentry | **Фильтрация** чувствительных полей в Sentry SDK; не логировать пароли/OTP |
| **Отсутствие алертов** | Атака не замечена | Алерты на 5xx, рост 401/429, аномалии регистраций |

### B.12. Социальная инженерия и учётки

| Угроза | Меры |
| :-- | :-- |
| Фишинг сотрудников | Обучение, политика паролей, МФА на почту/админ |
| Компрометация CI/облака | МФА на облако, минимальные IAM-роли |

## Часть C. Нормативное соответствие (152-ФЗ) — технический минимум

Сопоставление с типовыми мерами по ПП РФ № 1119 / ФСТЭК № 21 (формулировки в проекте уточняются при назначении уровня защищённости ИСПДн):

- **Идентификация и аутентификация** доступа к ПДн (роли, МФА для админов).
- **Управление доступом** (разграничение, принцип минимальных полномочий).
- **Ограничение программной среды** (контейнеры, отдельные сервисы).
- **Защита машинных носителей** (шифрование дисков/бэкапов).
- **Регистрация событий** (аудит админ-действий — `AuditLog` / simple-history по ТЗ).
- **Антивирусная защита** на серверах/рабочих станциях (по политике хостинга).
- **Обнаружение вторжений** (WAF/фаервол, алерты).
- **Контроль целостности** (хеши образов, контроль конфигов).

*Назначение уровня защищённости ИСПДн и включение в реестр — отдельное юридико-организационное решение; технически документ закладывает возможность соответствия.*

## Часть D. Конфигурация по слоям (эталон для реализации)

### D.1. Django / DRF (целевые настройки)

- `DEBUG = False` в проде; `ALLOWED_HOSTS` строго.
- `SECURE_SSL_REDIRECT`, `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`, `SECURE_HSTS_SECONDS` (>0), `SECURE_REFERRER_POLICY`, `SECURE_CONTENT_TYPE_NOSNIFF`, `X_FRAME_OPTIONS`.
- Пароли: хэшеры **Argon2** (предпочтительно) или **PBKDF2** по умолчанию Django.
- **CORS** — явный allow-list origin фронтенда, не `*`.
- **DRF:** `DEFAULT_AUTHENTICATION_CLASSES`, `DEFAULT_PERMISSION_CLASSES`, `DEFAULT_THROTTLE_CLASSES`.
- Валидация входных данных: сериализаторы + ограничение длины строк (DoS на JSON).

### D.2. Nginx (ориентир)

- `limit_req_zone` для API и отдельно для auth.
- `client_max_body_size` согласован с лимитом загрузки файлов.
- Заголовки: HSTS, X-Content-Type-Options, Referrer-Policy, CSP (постепенно ужесточать).
- Таймауты к upstream для защиты от slowloris.

### D.3. PostgreSQL

- Отдельный пользователь БД без прав **SUPERUSER**; минимальные GRANT.
- Резервное копирование: шифрование, тест восстановления.

### D.4. Секреты и окружения

- `base.py` / `dev.py` / `prod.py` по ТЗ; секреты только из **environment** или **secret manager**.
- `.env` в **.gitignore**; пример `.env.example` без секретов.

## Часть E. Чек-листы по этапам разработки (чтобы ничего не забыть)

### E.1. Перед началом спринта (для каждой фичи)

- [ ] Описана ли фича в терминах **данных** (какие ПДн, кто видит)?
- [ ] Есть ли **новые эндпоинты** — нужны ли rate limit и авторизация?
- [ ] Есть ли **состояние** (заказ, слот) — нужна ли **атомарность** / блокировка?
- [ ] Есть ли **внешний вызов** (SMS, оплата, гео) — SSRF/таймауты/ретраи?

### E.2. Code review (обязательные вопросы)

- [ ] Все запросы к объектам с владельцем фильтруются по **текущему пользователю**?
- [ ] Нет ли **сырого SQL** и **exec/eval**?
- [ ] Пользовательский ввод не попадает в **HTML** без экранирования?
- [ ] Файлы: тип, размер, имя файла — **валидированы**?
- [ ] Секреты не в коде?
- [ ] Логи не содержат **OTP, паролей, полных номеров карт**?

### E.3. Перед релизом в прод

- [ ] `DEBUG` выключен, секреты ротированы от dev.
- [ ] TLS и HSTS настроены.
- [ ] Бэкапы включены, **пробное восстановление** зафиксировано.
- [ ] Sentry (или аналог) с маскированием ПДн.
- [ ] Админка: скрытый URL + **МФА**.
- [ ] Зависимости проверены (`pip-audit` / аналог).

### E.4. После инцидента (шаблон действий)

1. Изоляция (отключение скомпрометированных ключей, блокировка учёток).
2. Сбор логов, фиксация временной шкалы.
3. Устранение причины, патч, ротация секретов.
4. Уведомления по **152-ФЗ** и пользователям — **по решению юриста** при утечке ПДн.
5. Postmortem и обновление чек-листов.

## Приложение: Быстрая матрица «Фича ТЗ → угрозы»

| Фича из ТЗ | Ключевые угрозы |
| :-- | :-- |
| SMS-авторизация | Перебор OTP, SMS-флуд, утечка кода в логах |

| Админка | Брутфорс, CSRF, утечка через массовые действия |
