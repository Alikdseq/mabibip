## 1. Назначение и границы проекта

### 1.1. Назначение

MVP обеспечивает:

- публичный **каталог СТО** с поиском и карточкой станции;
- **ручное** создание окон записи владельцем СТО;
- **запись клиента** в выбранное окно с подтверждением со стороны СТО;
- **отзывы** только после завершённого визита;
- **ручную монетизацию** (продление подписки администратором в админке).

### 1.2. Явно не входит в MVP (запрет на расширение в рамках этой итерации)

| Исключено | Причина |
| :-- | :-- |
| Чаты, WebSockets, Django Channels | Экономия времени |
| Elasticsearch, геопоиск на карте | Поиск по подстроке (название/адрес) |
| Redis, Celery, очереди | Синхронная отправка email; фон — только системный cron при необходимости |
| Vue.js / SPA | Фронт: Django Templates + HTMX + Alpine.js |
| Docker на старте | Ручной деплой на VPS |
| ЮKassa, Тинькофф, рекуррент | Ручные счета / перевод; учёт оплаты в админке |
| SMS, регистрация по телефону | Регистрация клиента по **email** |
| Автогенерация слотов по расписанию | Владелец добавляет окна вручную |
| Пуш-уведомления | Только email |

## 2. Технический стек

| Компонент | Выбор | Обоснование |
| :-- | :-- | :-- |
| Бэкенд | **Django 5** + **Django REST Framework** | Быстрый CRUD; DRF при необходимости для будущего API; в MVP основной UI — шаблоны |
| СУБД | **PostgreSQL 16** | Надёжность; расширение **PostGIS** установить, **запросы MVP без гео** |
| Кэш / брокер | **Нет** (без Redis) | Нагрузка порядка ~100 записей/день |
| Очереди | **Нет** (без Celery) | Email — синхронно; тяжёлые задачи не планируются |
| Фронтенд | **Django Templates** + **HTMX** + **Alpine.js** + **Bootstrap 5** | Минимум JS, интерактивность без SPA |
| Деплой | **VPS** (Timeweb / Beget и аналоги), **Gunicorn** + **Nginx** | `git pull`, `systemctl restart gunicorn` |
| Платежи | **Вне системы**: перевод на карту/рсч; в системе — поля подписки и ручное продление | Эквайринг — после MVP |

## 3. Роли и доступ

| Роль | Описание |
| :-- | :-- |
| **Клиент** | Регистрация по email; поиск СТО; запись; личный кабинет с бронированиями; отзыв после `completed` |
| **Владелец СТО** | Учётная запись создаётся **вручную** админом или по заявке + звонок; ЛК: заявки, слоты, статистика |
| **Администратор** | Полный доступ через **Django Admin**; модерация; продление подписки; создание владельцев СТО |

**Политика доступа владельцев СТО:** на старте не предусмотрена самостоятельная публичная регистрация владельца СТО «с улицы» — только после согласования/оплаты (ручной процесс).

## 4. Функциональные требования

### 4.1. Модуль «Регистрация и вход»

**Клиент**

- Регистрация: **email** (уникальный), пароль, подтверждение email **по ссылке из письма**.
- Вход: email + пароль.
- Сброс пароля: **только через email** (стандартный механизм Django).

**Владелец СТО**

- Создание пользователя с флагом владельца и привязкой к `ServiceStation` — через **Django Admin** (или внутренняя форма заявки без автовыдачи доступа — на усмотрение реализации, итог: доступ выдаёт админ).

### 4.2. Модуль «Каталог СТО» (публичная часть)

**Список СТО**

- Поля в карточке списка: название, адрес, **средний рейтинг** (среднее арифметическое оценок отзывов), индикатор **«Есть места сегодня»** (есть ли хотя бы одно свободное окно на текущую календарную дату).
- **Фильтр/поиск:** по **названию** и **адресу** — по подстроке (`icontains`), без карты и без PostGIS.

**Карточка СТО**

- До **5 фотографий** (загрузка через админку на старте для тестовых данных).
- Текстовое описание.
- Блок **отзывов** (только от пользователей с реальным завершённым бронированием — см. п. 4.6).
- Блок **«Ближайшая запись»:** сетка на **7 дней** (от текущей даты); выбор даты → список **свободных окон** с указанием **поста** (наименование `WorkBay`).

**Видимость в каталоге**

- В публичном списке и в поиске участвуют только СТО с **активной подпиской** и флагом активности (см. п. 4.7).

### 4.3. Модуль «Запись на сервис» (ядро)

**Сценарий клиента**

1. Выбор свободного окна (слота).
2. В модальном окне (HTMX/Alpine): **госномер**, **марка/модель** (одно поле), **описание проблемы**, **телефон для связи**.
3. Отправка формы — создание бронирования.

**Бэкенд**

- Проверка: слот ещё **свободен** (нет пересечения с другой бронью, статус слота допускает запись).
- Создание `Booking` со статусом **`pending_confirmation`** (в ТЗ ниже зафиксировано как `pending` — см. п. 6; единообразно использовать одно значение в коде; рекомендуется **`pending`** для краткости с последующим переименованием в UI).
- Слот переводится в состояние **зарезервирован**: недоступен для других; **удержание 30 минут** — если заявка не подтверждена СТО, слот снова становится доступным (без Celery: реализация через поле **`reserved_until`** на слоте и проверку при выдаче списка слотов / при создании брони, либо периодический `cron` + `manage.py` — зафиксировать в реализации).
- Страница успеха: текст в духе: *«Заявка отправлена. СТО должно подтвердить запись в течение часа»*.

**Уведомление СТО**

- Email владельцу: новая заявка с номером, датой/временем, ссылкой в ЛК.
- В ЛК СТО заявка визуально выделена (например, «мигает» / бейдж — реализация на Bootstrap + CSS/HTMX).

### 4.4. Модуль «Личный кабинет СТО»

**Дашборд**

- Заявки на **сегодня** и **завтра** (списки).
- Действия: **«Подтвердить»**, **«Отклонить»** (перевод в `canceled` или аналог), **«Завершить»** после оказания услуги → статус **`completed`** (только со стороны СТО).

**Управление слотами (упрощённо)**

- Форма **«Добавить окно»:** дата, время начала, время окончания, выбор **поста** (`WorkBay`).
- Без автогенерации по шаблону недели.

**Статистика**

- Количество записей за **текущий календарный месяц** (по подтверждённым/всем броням — уточнить в реализации; минимум: все созданные `Booking` за месяц или только не отменённые).

### 4.5. Модуль «Личный кабинет клиента»

- Список своих бронирований и статусов.
- После **`completed`** — одна ссылка/кнопка **«Оставить отзыв»** на бронь без отзыва.

### 4.6. Модуль «Отзывы»

- Отзыв доступен **только** если `Booking.status == completed`.
- **Один заказ — один отзыв** (`OneToOne` booking ↔ review).
- Поля: оценка **1–5**, текст.
- В админке видна связь `Booking` → `Review` для контроля.

### 4.7. Модуль «Монетизация» (ручной биллинг)

Поля у `ServiceStation`:

- `subscription_plan`: минимум **`free`**, **`basic`** (расширение enum допустимо).
- `subscription_paid_until`: дата **включительно** или «оплачено до» — зафиксировать в коде однозначно (рекомендация: **доступ до конца указанного дня** включительно).

**Правила**

- Если `subscription_paid_until` **меньше сегодняшней даты** (или NULL для платного тарифа — правило задать явно):
  - СТО **не отображается** в публичном каталоге и поиске.
  - При входе в ЛК СТО — **заглушка** «Оплатите подписку» (без онлайн-оплаты в MVP).
- Администратор вручную продлевает дату после получения оплаты (перевод на карту/рсч).

### 5.2. СТО, посты, слоты

```python
# stations/models.py
class ServiceStation(models.Model):
    owner = models.ForeignKey(User, on_delete=models.CASCADE, related_name='owned_stations')
    name = models.CharField(max_length=200)
    address = models.CharField(max_length=500)
    description = models.TextField(blank=True)
    subscription_plan = models.CharField(max_length=20, default='basic')  # free / basic
    subscription_paid_until = models.DateField(null=True, blank=True)
    is_active = models.BooleanField(default=True)

class WorkBay(models.Model):
    station = models.ForeignKey(ServiceStation, on_delete=models.CASCADE, related_name='bays')
    name = models.CharField(max_length=50)

class TimeSlot(models.Model):
    bay = models.ForeignKey(WorkBay, on_delete=models.CASCADE, related_name='slots')
    date = models.DateField()
    start_time = models.TimeField()
    end_time = models.TimeField()
    is_available = models.BooleanField(default=True)
    # Рекомендуется для MVP без Celery:
    # reserved_until = models.DateTimeField(null=True, blank=True)
```

**Связь брони и слота:** `Booking.slot` — **OneToOne** к `TimeSlot` (один слот — не более одной брони). При необходимости удержания 30 мин без финальной брони — доработать модель слота (см. п. 4.3).

### 5.3. Бронирование

```python
# bookings/models.py
class Booking(models.Model):
    STATUS_CHOICES = [
        ('pending', 'Ожидает подтверждения'),
        ('confirmed', 'Подтверждено'),
        ('completed', 'Завершено'),
        ('canceled', 'Отменено'),
    ]
    client = models.ForeignKey(User, on_delete=models.CASCADE, related_name='bookings')
    station = models.ForeignKey(ServiceStation, on_delete=models.CASCADE, related_name='bookings')
    slot = models.OneToOneField(TimeSlot, on_delete=models.CASCADE, related_name='booking')
    car_info = models.CharField(max_length=100)
    contact_phone = models.CharField(max_length=20)  # телефон из формы записи
    description = models.TextField()
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default='pending')
    created_at = models.DateTimeField(auto_now_add=True)
```

*Примечание: в исходном тексте ТЗ в форме указан телефон — поле `contact_phone` добавлено к эталону.*

## 6. Согласованность статусов и UI

- В пользовательском тексте допускается «ожидает подтверждения»; в БД единый код **`pending`**.
- Переходы: `pending` → `confirmed` | `canceled`; `confirmed` → `completed` | `canceled`; `completed` и `canceled` — терминальные (отзыв только из `completed`).

## 9. Критерии приёмки MVP

1. Клиент регистрируется по email, подтверждает почту, видит каталог только «оплаченных» СТО, открывает карточку, выбирает слот, создаёт заявку; слот недоступен другим.
2. Владелец СТО получает email о новой заявке; в ЛК подтверждает/отклоняет/завершает.
3. После `completed` клиент оставляет **один** отзыв; средний рейтинг на карточке пересчитывается.
4. При просроченной подписке СТО скрыто из каталога, ЛК СТО показывает заглушку; после ручного продления в админке снова видно.
5. Проект развёрнут на VPS, доступен по HTTPS (желательно).
