# План SEO-оптимизации «ПроМастер»

Документ для пошагового внедрения: один шаг — одна сессия работы. После каждого шага отмечайте статус (например, в коммите или чекбоксом локально).

**Контекст проекта:** Django + шаблоны, каталог `/sto/`, карточки `/sto/<slug>/`, фильтры через GET, словарь фраз `списокзапросов.txt`, услуги `списокуслуг.txt`, гео через сессию и `?city=`.

---

## Фаза A — Технический фундамент (индексация и дубли)

**Статус в коде:** реализовано (`/robots.txt`, `/sitemap.xml`, canonical в `base.html`, логика в `apps/core/seo.py`, карты в `apps/core/sitemaps.py`). После деплоя: `migrate` (приложение `sites`), задать `SITE_BASE_URL`, при необходимости `python manage.py sync_site_domain`.

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **A1** | Настроить прод-переменную `SITE_BASE_URL` (полный `https://домен` без слеша в конце) | В JSON-LD и будущих OG URL абсолютные и корректные |
| **A2** | Добавить отдачу `robots.txt` (исключить админку, кабинеты, служебные пути; указать URL sitemap) | `GET /robots.txt` отдаёт валидный файл, sitemap указан |
| **A3** | Реализовать `sitemap.xml` (индекс + отдельные карты или одна карта на MVP) | В карту попадают только канонические публичные URL (главная, каталог без мусорных параметров, карточки СТО) |
| **A4** | Ввести `rel="canonical"` на главной, каталоге (базовый URL без лишних UTM/параметров сортировки — политика зафиксирована в коде/доке), карточке СТО | В HTML есть один канонический URL на страницу; дубли параметров не конкурируют |
| **A5** | Политика пагинации каталога (`?page=`) | Решение задокументировано: либо canonical на страницу 1 с теми же фильтрами, либо self-canonical на каждую страницу — и реализовано |

---

## Фаза B — Мета, соцсети, базовая разметка

**Статус в коде:** главная и каталог задают `seo_meta_description` / `seo_og_title` в `HomePageView` и `StationListView`; Open Graph и Twitter Card в `templates/base.html`; запасное `og:image` — `static/pm-brand-sprite.svg` через контекст-процессор; карточка СТО — `@graph` (BreadcrumbList + AutoRepair/Person) в `apps/stations/detail_schema.py`, телефон и фото в разметке при наличии.

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **B1** | Уникальный `<meta name="description">` для главной | Длина ~150–160 символов, гео и УТП |
| **B2** | Динамические title/description для каталога с учётом фильтров (город, услуга, марка — в разумных пределах) или минимум уникальный description при активном `city` / `service` | Нет одного и того же сниппета для всех выдач каталога |
| **B3** | Open Graph: `og:title`, `og:description`, `og:url`, `og:type`, `og:image` (общая картинка бренда на старте) | Валидный превью-шаринг; `og:url` совпадает с canonical |
| **B4** | JSON-LD `BreadcrumbList` на карточке СТО (и при появлении — на других типах страниц) | Проверка в schema.org validator без критичных ошибок |
| **B5** | Проверка существующего JSON-LD (`AutoRepair` / `Person`): телефон, часы (если есть в модели), `image` при наличии фото | Данные из БД, без пустых обязательных полей там, где тип их ждёт |

## Фаза C — Структура URL и лендинги под семантику

**Статус в коде:** `path("", include("apps.stations.landing_urls"))` — `/uslugi/<slug>/` (категория), `/marki/<slug>/` (марка); шаблоны `service_category_landing.html`, `car_brand_landing.html`; 301 с `/sto/?service=<slug>` при отсутствии других фильтров — `catalog_redirect.py`; ссылки с главной, подсказок поиска и карточки СТО ведут на лендинги; sitemap секции `services`, `brands`. **C4** (хабы городов) не делались — при мультигороде позже.

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **C1** | Спроектировать ЧПУ для услуг (например `/uslugi/<slug>/`) на базе `ServiceCategory` | Один канонический URL на категорию; редирект 301 с `?service=<slug>` на ЧПУ (опционально, но желательно) |
| **C2** | Страница категории услуги: H1, лид-текст, блок «похожие услуги», кнопка/форма в каталог с пресетом `service` или `cat` | Пользователь и бот видят уникальный контент, не пустой шаблон |
| **C3** | Страницы марок `/marki/<slug>/` (на базе `CarBrand`) | Топ услуг по марке, внутренние ссылки на каталог с `brand=` |
| **C4** | (Опционально) Городские хабы `/gorod/<slug>/` при мульти-городе | Уникальный текст и список районов/ссылок; иначе отложить до расширения гео |
| **C5** | Внутренняя перелинковка: главная → услуги → каталог; карточка СТО → категории | Нет «тупиковых» страниц без исходящих ссылок |

## Фаза D — Контент и словарь

**Статус в коде/доках:** **D1** — runbook `docs/seo/runbook-search-phrases.md` (`import_search_dictionary`, `ensure_search_phrases`). **D2** — поля `ServiceCategory.landing_lead`, `landing_faq`, пресеты `apps/stations/seed_data/category_landing_presets.py` (20 категорий), команда `python manage.py seed_category_landing_content`, вывод в `templates/stations/service_category_landing.html`, FAQ JSON-LD в `apps/stations/landing_seo.py`. **D3** — расширенные title/description для «горячих» слагов через `build_service_category_seo` и `HOT_SERVICE_SLUGS`. **D4** — план статей `docs/seo/article-outlines.md` (публикация в CMS — вне репозитория).

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **D1** | Импорт/актуализация `ServiceSearchPhrase` из `списокзапросов.txt` на проде | Команда `import_search_dictionary` / `ensure_search_phrases` отработана в CI или runbook |
| **D2** | Привязка текстов услуг из `списокуслуг.txt` к полям БД (краткое описание, FAQ) или к шаблонам лендингов | Минимум топ-20 услуг с уникальным абзацем + 3–5 FAQ |
| **D3** | Расширение семантики: шаблоны фраз «{услуга} Владикавказ», «цена», «запись онлайн» в title/description лендингов | Покрыты высокочастотные категории из `homepage.HOT_SERVICE_TILES` и аналоги |
| **D4** | Блог/гайды (если в MVP): 5–10 статей под симптомы из словаря | Каждая статья ссылается на 1–2 лендинга услуг и каталог |

## Фаза E — Локальное SEO и внешние сигналы

**Статус в доках:** **E1** `docs/seo/sto-local-profiles.md`, **E2** `docs/seo/review-outreach.md`, **E3** `docs/seo/crowd-partnerships.md`. Трекинг крауда — по E3, вне репозитория.

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **E1** | Методичка для СТО: Яндекс Бизнес, Google Business Profile, единый NAP | Файл в `docs/seo/` или рассылка партнёрам |
| **E2** | Стратегия отзывов (после визита): напоминание со ссылкой на карточку | Не нарушает правила платформы и 152-ФЗ |
| **E3** | Крауд/партнёрства: список площадок и формат ссылок (на лендинги, не только главную) | Таблица трекинга (достаточно вне репозитория) |

## Фаза F — Аналитика и контроль

**Статус в доках:** **F1** `docs/seo/webmaster-gsc-f1.md` (Вебмастер + GSC, URL sitemap, предусловия `SITE_BASE_URL` / `sync_site_domain`). **F2** `docs/seo/indexing-monitor-f2.md` (ответственный, еженедельный чеклист первый месяц). **F3** `docs/seo/lighthouse-cwv-f3.md` (Lighthouse / PSI, базовая таблица метрик, проверка перед релизом).

| Шаг | Задача | Критерий готовности |
|-----|--------|---------------------|
| **F1** | Подключение домена к Яндекс.Вебмастер и Google Search Console | Подтверждение владения, загрузка sitemap |
| **F2** | Мониторинг индексации и ошибок сканирования (еженедельно первый месяц) | Ответственный и чеклист |
| **F3** | Lighthouse / Core Web Vitals на главной и каталоге (mobile) | Зафиксированы базовые метрики; регрессии ловятся перед релизом |

## Порядок выполнения (рекомендуемый)

1. **A1 → A2 → A3** — чтобы поисковики видели границы сайта и страницы.  
2. **A4 → A5** — убрать риск дублей.  
3. **B1 → B2 → B3** — улучшить CTR в выдаче.  
4. **B4 → B5** — усилить сниппеты и понимание сущностей.  
5. **C1 → C2** — главный SEO-прирост по коммерческим запросам.  
6. **C3 → C5**, затем **D** и **E**, параллельно **F1**; после подтверждения домена — **F2** (ритм) и **F3** (базовая линия CWV).
